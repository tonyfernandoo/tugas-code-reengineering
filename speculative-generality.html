Speculative Generality


  • Apa Itu Speculative Generality?

    • Speculative Generality, adalah salah satu jenis code smell yang terjadi ketika terdapat kelas, modul, atau fungsi dalam sebuah program yang dirancang untuk menangani banyak kasus yang mungkin terjadi di masa depan, namun pada kenyataannya, kebutuhan tersebut tidak pernah muncul atau terjadi. Dalam kata lain, Speculative Generality terjadi ketika sebuah komponen perangkat lunak lebih kompleks dari yang seharusnya, karena mencoba memprediksi kebutuhan yang mungkin muncul di masa mendatang, tanpa adanya bukti konkret bahwa kebutuhan tersebut akan muncul.

    Keberadaan Speculative Generality dapat mengakibatkan kompleksitas yang tidak perlu dalam kode sumber, meningkatkan risiko kesalahan, dan membuat pemeliharaan kode menjadi lebih sulit. Selain itu, komponen perangkat lunak yang terlalu generik atau umum juga dapat sulit dipahami oleh pengembang baru atau pengembang yang tidak terlibat dalam pengembangan awalnya.

    Dengan memahami Speculative Generality, pengembang dapat lebih berhati-hati dalam merancang komponen perangkat lunak dan menghindari jebakan kompleksitas yang tidak perlu.


  • Tanda-tanda Speculative Generality

    • Bagaimana kita tahu jika kode kita terdapat adanya Speculative Generality? Berikut adalah beberapa tanda yang dapat membantu Anda mengidentifikasi keberadaannya:

    1. Kelas atau Modul yang Terlalu Umum: Perhatikan kelas atau modul yang merangkum banyak kasus yang mungkin terjadi di masa depan, namun pada kenyataannya, hanya sebagian kecil dari kasus tersebut yang terjadi. Contoh kelas atau modul seperti ini sering kali memiliki banyak opsi parameter atau metode yang tidak selalu digunakan dalam setiap kasus penggunaan.

    2. Fitur atau Fungsionalitas yang Tidak Digunakan: Tinjau fitur atau fungsionalitas yang telah dirancang untuk menangani situasi yang belum pernah terjadi atau kebutuhan yang belum tentu muncul. Jika fitur-fitur ini jarang digunakan atau bahkan tidak pernah dipanggil, mungkin ada tanda Speculative Generality.

    3. Komponen yang Sulit Dipahami: Jika terdapat kelas atau modul yang terlalu kompleks atau sulit dipahami karena mencoba menangani banyak kasus yang mungkin terjadi di masa depan, ini bisa menjadi tanda Speculative Generality. Kompleksitas yang berlebihan dalam desain sering kali menunjukkan adanya upaya untuk memprediksi kebutuhan yang belum tentu muncul.

    Dengan memperhatikan tanda-tanda ini, Anda dapat mengidentifikasi Speculative Generality dalam kode sumber Anda dan mengambil langkah-langkah yang diperlukan untuk membersihkannya. Dengan mempertahankan kode yang bersih dan efisien, Anda dapat meningkatkan keterbacaan, pemeliharaan, dan skalabilitas dari proyek perangkat lunak Anda.


  • Strategi untuk Mengatasi Speculative Generality

    • Mengatasi Speculative Generality dalam kode sumber merupakan langkah penting untuk menjaga kebersihan, keterbacaan, dan kualitas dari proyek perangkat lunak Anda. Berikut adalah beberapa strategi yang dapat Anda terapkan untuk mengatasi Speculative Generality:

    1. Kembangkan dengan Keperluan Saat Ini: Fokuskan perancangan dan implementasi kelas atau modul pada kebutuhan saat ini, bukan pada spekulasi tentang apa yang mungkin diperlukan di masa mendatang. Hindari menambahkan fitur atau fungsionalitas yang tidak diperlukan saat ini dan tetapkan prinsip bahwa "tidak ada spekulasi tanpa bukti".

    2. Refaktorisasi Kode yang Terlalu Umum: Tinjau kembali kelas atau modul yang terlalu umum dan refaktorlah agar menjadi lebih spesifik sesuai dengan kebutuhan saat ini. Pisahkan fungsionalitas yang tidak relevan atau jarang digunakan ke dalam kelas atau modul yang terpisah, atau hapuslah sepenuhnya jika tidak diperlukan.

    3. Fokus pada Keterbacaan dan Keterartikan: Desainlah kelas atau modul dengan fokus pada keterbacaan dan keterartikan. Buatlah kode yang mudah dipahami dan mudah dimodifikasi oleh pengembang lain dengan menghindari kompleksitas yang tidak perlu.

    4. Pemantauan Ketergantungan Antar Komponen:Perhatikan ketergantungan antar kelas atau modul dalam proyek Anda. Identifikasi dan evaluasi ketergantungan yang tidak perlu atau berlebihan dan ambil langkah-langkah untuk memecahnya menjadi bagian yang lebih kecil dan lebih fokus.

    Dengan menerapkan strategi ini secara konsisten, Anda dapat mengatasi Speculative Generality dalam kode sumber Anda dan memastikan bahwa proyek perangkat lunak Anda tetap efisien, mudah dipelihara, dan sesuai dengan kebutuhan saat ini tanpa membebani dengan kompleksitas yang tidak perlu.

    • Salah satu prinsip extreme programming (XP) yang terkenal adalah You arenâ€™t gonna need it (YAGNI). Disarankan oleh XP untuk tidak menambahkan fungsionalitas sampai benar-benar diperlukan.

    Contohnya, pada package before.

    Requirement dari client adalah Price bisa memiliki Currency antara IDR atau USD.

    Programmer berspekulasi bahwa IDR dan USD adalah kurs jenis tradisional, nantinya akan ada kurs jenis digital seperti bitcoin. Oleh karena itu, Programmer membuat hirarki seperti di dalam package before.

    Before

    Currency.java :

    
package fowler.dispensables.speculative_generality.before;

public abstract class Currency {
	public abstract String getCode();
}

    Digital.java :

    
package fowler.dispensables.speculative_generality.before;

public abstract class Digital extends Currency {

}

    IDR.java :

    
package fowler.dispensables.speculative_generality.before;

public class IDR extends Traditional {
	@Override
	public String getCode() {
		return "IDR";
	}
}

    Price.java :

    
package fowler.dispensables.speculative_generality.before;

public class Price {
	int value;
	Currency currency;
	
	public Price(int value, Currency currency) {
		this.value = value;
		this.currency = currency;
	}
	
	public int getValue() {
		return value;
	}
	public void setValue(int value) {
		this.value = value;
	}
	public Currency getCurrency() {
		return currency;
	}
	public void setCurrency(Currency currency) {
		this.currency = currency;
	}
}

    Traditional.java :

    
package fowler.dispensables.speculative_generality.before;

public abstract class Traditional extends Currency {

}

    USD.java :

    
        package fowler.dispensables.speculative_generality.before;

        public class USD extends Traditional {
          @Override
          public String getCode() {
            return "USD";
          }
        }

    After

    Currency.java :

    
package fowler.dispensables.speculative_generality.after;

public abstract class Currency {
	public abstract String getCode();
}

    IDR.java :

    
package fowler.dispensables.speculative_generality.after;

public class IDR extends Currency {
	
	@Override
	public String getCode() {
		return "IDR";
	}
}

    Price.java :

    
package fowler.dispensables.speculative_generality.after;

public class Price {
	int value;
	Currency currency;
	
	public Price(int value, Currency currency) {
		this.value = value;
		this.currency = currency;
	}
	
	public int getValue() {
		return value;
	}
	public void setValue(int value) {
		this.value = value;
	}
	public Currency getCurrency() {
		return currency;
	}
	public void setCurrency(Currency currency) {
		this.currency = currency;
	}
}

    USD.java :

    
package fowler.dispensables.speculative_generality.after;

public class USD extends Currency {

	@Override
	public String getCode() {
		return "USD";
	}
}


    Dilakukan Collapse Hierarchy. Class Traditional dan Digital dihapus. class USD dan IDR menjadi turunan langsung dari class Currency.
