Divergent Change


  • Apa Itu Divergent Change?

    • Divergent Change, seperti namanya, adalah keadaan di mana sebuah kelas atau modul dalam kode kita mulai berubah secara terpisah, menjauh dari tujuan atau tanggung jawab awalnya. Ini seperti jalan bercabang dalam pengembangan perangkat lunak, di mana perubahan yang seharusnya terkait menjadi terpisah dan bergerak ke arah yang berlawanan.

    Ketika Divergent Change terjadi, tidak lagi terdapat alur logis yang konsisten dalam evolusi kode kita. Sebuah entitas kode mungkin akan berubah karena berbagai alasan yang tidak terkait, seperti permintaan fitur baru, perubahan kebijakan, atau perbaikan bug. Hal ini mengakibatkan fragmentasi dan kompleksitas yang tidak terkendali, menghasilkan kode yang sulit dimengerti dan dipelihara.

    Mengidentifikasi Divergent Change adalah langkah awal yang penting dalam membersihkan dan merawat basis kode kita. Dengan memahami esensi dari code smell ini, kita dapat memulai perjalanan menuju kode yang lebih bersih, terkelola, dan mudah dipahami.


  • Tanda-tanda Divergent Change

    • Bagaimana kita tahu jika kode kita mengalami Divergent Change? Berikut adalah beberapa tanda yang dapat membantu Anda mengidentifikasi keberadaannya:

    1. Banyaknya Perubahan: Jika Anda melihat sebuah kelas atau modul mengalami banyak perubahan dalam periode waktu yang singkat, ini bisa menjadi tanda Divergent Change. Perubahan-perubahan tersebut mungkin tidak terkait satu sama lain dan terjadi karena alasan yang berbeda.

    2. Perubahan yang Tidak Terkait: Divergent Change terjadi ketika perubahan-perubahan tersebut tidak terkait dengan satu fitur atau kebutuhan bisnis tertentu. Sebagai gantinya, mereka mungkin berkaitan dengan permintaan fitur baru, perubahan kebijakan, atau peningkatan kinerja.

    3. Kesulitan Memahami Kode: Divergent Change dapat membuat kode menjadi sulit dimengerti. Anda mungkin merasa kesulitan untuk melacak alur logis dari perubahan-perubahan tersebut dan memahami dampaknya terhadap kode yang ada.

    Dengan memahami tanda-tanda ini, Anda dapat lebih mudah mengidentifikasi keberadaan Divergent Change dalam kode Anda dan mengambil langkah-langkah untuk mengatasinya.


  • Strategi untuk Mengatasi Divergent Change

    • Divergent Change bukan hanya sekadar code smell yang mengganggu, tetapi juga memiliki dampak negatif yang signifikan pada proses pengembangan perangkat lunak. Berikut adalah beberapa dampak negatif yang dapat terjadi:

    1. Kompleksitas Kode yang Meningkat: Divergent Change menyebabkan peningkatan kompleksitas kode dengan memperkenalkan cabang-cabang yang tidak terkendali dalam evolusi kode. Hal ini membuat kode menjadi lebih sulit dimengerti dan dikelola.

    2. Menurunkan Keterbacaan dan Pemahaman Kode: Ketika kode mengalami Divergent Change, alur logisnya menjadi terputus dan sulit diikuti. Hal ini membuat kode sulit dipahami oleh pengembang lain yang harus memelihara atau berkolaborasi dalam pengembangan.

    3. Memperburuk Kualitas dan Keberlanjutan Kode: Divergent Change dapat mengarah pada kode yang kurang berkualitas dan tidak stabil. Kode yang sering berubah secara terpisah cenderung memiliki lebih banyak bug dan kesalahan, serta sulit dipelihara dalam jangka panjang.

    4. Meningkatkan Risiko Kesalahan dan Kegagalan:Kode yang mengalami Divergent Change memiliki risiko yang lebih tinggi untuk mengalami kesalahan dan kegagalan. Perubahan yang tidak terkendali dapat menyebabkan efek samping yang tidak diinginkan atau mengganggu fungsi-fungsi yang ada.

    Dengan memahami dampak negatif dari Divergent Change, kita dapat menyadari pentingnya untuk mengidentifikasi dan mengatasi code smell ini dalam pengembangan perangkat lunak.


  • Contoh Kasus

    • Dalam aplikasi manajemen toko sederhana yang ditulis dalam bahasa pemrograman Java, kita memiliki kelas "Product" yang bertanggung jawab untuk merepresentasikan produk yang dijual di toko. Namun, seiring dengan perkembangan aplikasi, kelas ini mulai mengalami Divergent Change karena berbagai alasan yang berbeda.

    Mari kita lihat contoh kode untuk kelas "Product":

    Before
    
public class Product {
    private String name;
    private String description;
    private double price;
    private List<Review> reviews; // Daftar ulasan produk

    // Konstruktor, getter, dan setter lainnya
}

    Kelas "Product" awalnya dibuat dengan atribut dasar seperti nama, deskripsi, dan harga. Namun, seiring waktu, muncul permintaan fitur baru seperti penambahan ulasan produk, yang mengakibatkan kelas ini mengalami banyak perubahan. Setiap kali ada permintaan fitur baru atau perubahan kebijakan, kelas Product harus diubah, menyebabkan fragmentasi dalam evolusi kode. Hal ini membuat kode yang terkait dengan Product sulit dipahami oleh pengembang lain dalam tim karena alur logisnya terputus, menyulitkan prediksi efek dari perubahan yang dilakukan.


    Kita akan memisahkan tanggung jawab untuk mengelola review ke dalam kelas terpisah yang disebut "ProductReviews."

    After

    Main.java :

    
import java.util.List;

public class Product {
  private String name;
  private String description;
  private double price;
  private ProductReviews reviews; // Objek untuk mengelola ulasan produk
                  
// Konstruktor, getter, dan setter lainnya
                  
// Method untuk menambah ulasan produk
public void addReview(Review review) {
    reviews.addReview(review);
  } 
                  
// Method lainnya yang terkait dengan produk
}

class ProductReviews {
  private List
}

    ProductReviews.java:

    
import java.util.ArrayList;
import java.util.List;

public class ProductReviews {
    private List
}

    Review.java:

    
import java.util.ArrayList;
import java.util.List;

public class Review {
    private String reviewerName;
    private String comment;
    private int rating;

    // Konstruktor, getter, dan setter lainnya
}


    Dengan melakukan extract class, kita memisahkan tanggung jawab untuk mengelola ulasan produk ke dalam kelas "ProductReviews.", yang membuat kelas "Product" tetap fokus pada tanggung jawab utamanya, yaitu merepresentasikan produk yang dijual di toko. Ini membantu mengatasi Divergent Change dengan mengelompokkan perubahan-perubahan terkait dengan ulasan produk ke dalam satu tempat, meningkatkan keterbacaan dan pemeliharaan kode.
