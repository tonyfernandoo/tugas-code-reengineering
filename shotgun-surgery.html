Shotgun Surgery


  • Apa Itu Shotgun Surgery?

    • Shotgun Surgery , adalah sebuah code smell yang terjadi ketika satu perubahan kecil dalam kode mengharuskan kita melakukan sejumlah besar perubahan di berbagai tempat dalam kode tersebut. Analoginya seperti menggunakan senapan patah untuk melakukan serangkaian tembakan kecil yang tersebar luas.

    Code smell ini mencerminkan kurangnya pemisahan tanggung jawab dalam struktur kode kita, sehingga setiap perubahan kecil berdampak besar pada berbagai bagian sistem. Shotgun Surgery sering kali menjadi tanda bahwa kita tidak mematuhi prinsip Single Responsibility Principle (SRP), yang menyarankan agar setiap kelas atau modul memiliki satu dan hanya satu alasan untuk berubah. Dengan memahami esensi dari Shotgun Surgery, kita dapat mulai mengidentifikasi dan mengatasi pola-pola ini dalam kode kita, meningkatkan kualitas, keterbacaan, dan keberlanjutan kode dalam jangka panjang.


  • Tanda-tanda Shotgun Surgery

    • Bagaimana kita tahu jika kode kita mengalami Shotgun Surgery? Berikut adalah beberapa tanda yang dapat membantu Anda mengidentifikasi keberadaannya:

    1. Perubahan Berantai: Jika perubahan kecil dalam kode mengharuskan kita melakukan perubahan di banyak lokasi yang berbeda, ini bisa menjadi tanda Shotgun Surgery. Kita mungkin menemukan diri kita berpindah-pindah antara berbagai kelas atau modul untuk melakukan perubahan yang seharusnya terkonsentrasi dalam satu tempat.

    2. Kode Duplikat: Adanya kode duplikat yang tersebar luas di berbagai bagian kode juga bisa menjadi tanda Shotgun Surgery. Ketika kita menemukan bahwa kita harus mengubah kode yang sama di banyak tempat untuk melakukan perubahan tertentu, hal ini menunjukkan bahwa kita mungkin menghadapi Shotgun Surgery.

    3. Fragilitas Kode: Shotgun Surgery sering kali meningkatkan fragilitas kode, di mana perubahan kecil dapat dengan mudah menyebabkan bug atau kegagalan di berbagai bagian sistem. Jika kita merasa bahwa kode kita rentan terhadap perubahan kecil yang tidak terduga, ini bisa menjadi tanda Shotgun Surgery.

    Dengan memahami tanda-tanda ini, kita dapat lebih mudah mengidentifikasi keberadaan Shotgun Surgery dalam kode kita dan mengambil langkah-langkah untuk mengatasi pola-pola ini guna meningkatkan kualitas dan keberlanjutan kode.


  • Strategi untuk Mengatasi Shotgun Surgery

    • Refactoring adalah praktik penting dalam mengatasi Shotgun Surgery dalam kode. Dengan melakukan beberapa teknik refactoring seperti "Move Method", "Move Field", dan "Inline Class" , kita dapat mengurangi ketergantungan antar kelas dan memisahkan tanggung jawab dengan lebih baik. Berikut adalah beberapa langkah refactoring yang dapat kita terapkan:

    1. Move Method: Dengan teknik ini, kita dapat memindahkan suatu metode yang sering berubah bersama-sama ke dalam kelas yang lebih sesuai. Dengan demikian, kita mengurangi dampak perubahan kecil pada banyak kelas lainnya.

    2. Move Field: Jika sebuah atribut sering diakses oleh metode-metode dalam kelas lain, pertimbangkan untuk memindahkannya ke kelas yang lebih tepat. Dengan cara ini, kita mengurangi ketergantungan dan memperbaiki desain yang rentan terhadap Shotgun Surgery.

    3. Inline Class: Jika sebuah kelas tidak lagi memiliki tanggung jawab yang cukup untuk berdiri sendiri, kita dapat mempertimbangkan untuk "inline" atau menggabungkannya kembali ke kelas yang menggunakan fungsionalitas tersebut. Ini membantu mengurangi ketergantungan yang tidak perlu antar kelas.

    Dengan menerapkan strategi-strategi refactoring ini secara bijaksana, kita dapat merapikan kode kita, meningkatkan pemisahan tanggung jawab, dan mengurangi risiko dari Shotgun Surgery. Hal ini membantu menjaga kode kita tetap bersih, terkelola dengan baik, dan mudah dipahami oleh tim pengembangan.


  • Contoh Kasus

    • Kita memiliki dua kelas, "Order" dan "Customer", yang terlibat dalam proses pembuatan pesanan. Setiap kali sebuah pesanan dibuat, kita perlu memperbarui informasi terkait pelanggan. Namun, proses untuk melakukan pembaruan ini tersebar di berbagai metode dalam kedua kelas tersebut, yang menghasilkan Shotgun Surgery.

    Before
    
class Order {
    private Customer customer;

    public void createOrder() {
        // Logika untuk membuat pesanan
        updateCustomerInfo();
    }

    private void updateCustomerInfo() {
        customer.updateOrderHistory();
        customer.updatePoints();
    }
}

class Customer {
    public void updateOrderHistory() {
        // Logika untuk memperbarui riwayat pesanan
    }

    public void updatePoints() {
        // Logika untuk memperbarui poin pelanggan
    }
  }

    Dalam kode di atas, proses pembaruan informasi pelanggan ("updateOrderHistory" dan "updatePoints") terjadi di dalam metode "updateCustomerInfo" di kelas "Order". Namun, metode-metode ini seharusnya menjadi tanggung jawab dari kelas "Customer". Hal ini menyebabkan Shotgun Surgery karena setiap kali kita ingin memperbarui informasi pelanggan, kita harus melakukan perubahan di berbagai tempat dalam kode.


    Kita dapat menggunakan teknik "Move Method" untuk memindahkan metode "updateOrderHistory" dan "updatePoints" dari kelas "Customer" ke kelas "Order" , sehingga proses pembaruan informasi pelanggan terpusat dalam satu tempat.

    After
    
class Order {
  private Customer customer;
                  
  public void createOrder() {
      // Logika untuk membuat pesanan
      customer.updateOrderInfo();
    }
  }
                  
class Customer {
  public void updateOrderInfo() {
      updateOrderHistory();
      updatePoints();
  }
                  
  private void updateOrderHistory() {
      // Logika untuk memperbarui riwayat pesanan
  }
                  
  private void updatePoints() {
      // Logika untuk memperbarui poin pelanggan
  }
}


    Setelah menggunakan teknik "Move Method" , metode "updateOrderHistory" dan "updatePoints" sekarang menjadi bagian dari metode "updateOrderInfo" di kelas "Customer". Ini menghilangkan Shotgun Surgery karena pembaruan informasi pelanggan terpusat dalam satu tempat di dalam kelas "Customer", dan kita tidak perlu lagi menyebarkan perubahan ke berbagai tempat dalam kode.
